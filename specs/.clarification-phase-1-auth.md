## Critical Issues (Block Implementation) ⛔

### 🔴 Issue 1: Supabase Auth Integration Ambiguity

**Problem**: Specification mentions "Supabase Auth" for password hashing but is unclear on architecture:

- Does the system call Supabase Auth API directly OR use local bcrypt?
- Where is the password stored? (Supabase Auth table vs. custom users table?)
- How is password verification handled? (Supabase delegates OR manual verification?)

**Current Text**:
```
FR-002: System MUST hash passwords using bcrypt (or Supabase Auth default) before storing
```

**Confusion**:
- "or Supabase Auth default" → Which one? Different implementations!
- If Supabase Auth is used, then `SupabaseAuthClient.CreateUser()` handles everything
- If local bcrypt, then we need a separate password storage mechanism

**Impact**: 
- ❌ Cannot implement `SupabaseAuthClient` without knowing this
- ❌ Cannot design user table schema without knowing this
- ❌ Security implications differ significantly

**Recommendation**:
```
DECISION REQUIRED: Pick one approach:

Option A (Recommended - Simpler): 
  Use Supabase Auth native service
  - Password stored in Supabase Auth (not in users table)
  - Call Supabase Auth API for verification
  - JWT token issued by Supabase

Option B (More Control):
  Use custom password storage
  - Hash password locally with bcrypt
  - Store in separate password table
  - Implement JWT generation locally

RECOMMEND: Option A (reduces complexity)
```

### 🔴 Issue 2: JWT Token Generation vs. Supabase Issuance

**Problem**: Unclear who generates the JWT token:

**Current Specs Say**:
- "FR-003: System MUST issue JWT token on successful login" → Implies we generate it
- "Infrastructure: SupabaseAuthClient.VerifyCredentials(email, password)" → Supabase verifies
- "JwtTokenService generates JWT token if credentials valid" → We generate it
- BUT: "Supabase JWT Secret (environment variable)" in config → Supabase owns the secret

**Confusion**:
```
Scenario A: We call Supabase Auth, get JWT from Supabase
Scenario B: We call Supabase to verify, then generate our own JWT using Supabase's secret
Scenario C: Supabase does everything, we just pass through

Which one?
```

**Impact**:
- ❌ Different implementations of `JwtTokenService`
- ❌ Different token validation flow
- ❌ Different claims structure possible
- ❌ Different refresh token handling

**Recommendation**:
```
DECISION REQUIRED:

Option A (Recommended - Supabase Native):
  Use Supabase Auth API directly
  - Call POST /auth/v1/token with email/password
  - Supabase returns JWT + refresh token
  - We store in HttpOnly cookie
  - Skip JwtTokenService, use Supabase response directly

Option B (Hybrid - Local Generation):
  Verify with Supabase, generate JWT locally
  - Call Supabase to verify credentials
  - Generate JWT locally using Supabase JWT secret
  - Implement custom JwtTokenService
  - More complex, same security

RECOMMEND: Option A (less custom code, proven by Supabase)
```

### 🟡 Issue 4: Session Timeout & Token Refresh Ambiguity

**Problem**: JWT lifetime is 24 hours, but UX implications unclear:

**Not Specified**:
- What happens after token expires during active session?
  - Silent refresh? (Need refresh token in spec)
  - Force logout? (Bad UX for long-running apps)
  - Extended session? (Security risk)
  
- Is there a "remember me" feature?
  - If yes: longer token lifetime?
  - If no: user must re-login every 24 hours

**Current Text**:
```
JWT_EXPIRY_HOURS=24
```

**Confusion**:
- User session > 24 hours without activity → auto logout
- User on dashboard for 20 hours → token expires, then what?
- No refresh token mechanism mentioned

**Recommendation**:
```
CLARIFY:

Q1: Token refresh strategy?
    A: DECISION - One of:
       - No refresh: User logs out after 24h (simplest)
       - With refresh: Sliding window, refresh token in secure cookie (best UX)
       - Extended: Long-lived tokens (security risk, not recommended)

RECOMMEND: "No refresh" for Phase 1 simplicity
  - 24h expiry
  - User logs out after 24h inactivity
  - "Session expired" redirect to login
```

### 🟡 Issue 5: "Weak Password" Definition Ambiguity

**Problem**: Password requirements conflict:

**Current Specs Say**:
- FR-001: "password (minimum 8 characters)" ← Only length specified
- Error Handling: "Password must include upper, lower, digit, special char" ← Complexity implied
- Security: "Complexity optional but recommended" ← Optional = not required

**Contradiction**:
```
Is password "password1" valid?
  - 8+ chars? YES
  - Includes upper, lower, digit? NO
  - Complexity required? UNCLEAR

"Optional but recommended" → Do we enforce or not?
```

**Recommendation**:
```
CLARIFY ONE OF:

Option A (Strict - Recommended for security):
  Password MUST:
  - Be 8+ characters
  - Include uppercase letter
  - Include lowercase letter
  - Include digit
  - Include special character (!@#$%^&*)

Option B (Lenient - Better UX):
  Password MUST:
  - Be 8+ characters
  
  System SHOULD warn if missing uppercase/lowercase/digit

RECOMMEND: Option A (security for expense data)
  Update spec: FR-001 → "password (minimum 8 characters, must include upper, lower, digit, special)"
```
