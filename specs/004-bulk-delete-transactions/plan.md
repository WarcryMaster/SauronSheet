# Implementation Plan: Bulk Delete Transactions

**Branch**: `004-bulk-delete-transactions` | **Date**: 2026-03-13 | **Spec**: [`specs/004-bulk-delete-transactions/spec.md`](spec.md)

## Summary

Enable users to select multiple transactions via checkboxes and delete them atomically with a safety confirmation dialog. Reduces deletion time from 2-3 minutes (one-by-one) to <30 seconds for 5+ transactions. Implementation spans Application layer (MediatR command + handler), Infrastructure layer (repository delete method), and Frontend (UI components for selection and confirmation).

---

## Technical Context

**Language/Version**: C# / .NET 10  
**Primary Dependencies**: MediatR 12+, Supabase Postgrest client (C#), Razor Pages  
**Storage**: Supabase PostgreSQL (transactions table)  
**Testing**: xUnit + Moq (unit/integration), in-memory Supabase doubles  
**Target Platform**: Web (Razor Pages + JavaScript)  
**Project Type**: Full-stack .NET Core web application  
**Performance Goals**: Delete 5+ transactions in <2 seconds server-side  
**Constraints**: Atomic deletion (all succeed or all rollback); multi-tenant isolation mandatory  
**Scale/Scope**: Supports 0-10,000 transactions per user; bulk select limited to 1000 max

## Implementation Layers

### Layer 1: Domain (NO CHANGES)
- Transaction aggregate root already exists with TransactionId, UserId
- No new domain logic required (validation at Application layer)

### Layer 2: Application — Command + Handler

**BulkDeleteTransactionsCommand.cs**
```csharp
public record BulkDeleteTransactionsCommand(
    UserId UserId,
    IReadOnlyList<TransactionId> TransactionIds
) : IRequest<BulkDeleteResultDto>;
```

**BulkDeleteTransactionsHandler.cs**
- Input validation:
  - Reject if TransactionIds is empty or exceeds 1000 items
  - Reject if any TransactionId is Guid.Empty
- Multi-tenancy check:
  - Load all transactions from repository (scoped to UserId)
  - Verify loaded count == requested count (detect attempted cross-tenant access)
  - Throw DomainException if mismatch
- Deletion:
  - Call repository.DeleteTransactionsByIdsAsync(UserId, TransactionIds)
  - Catch database exceptions → return result with error message
- Response:
  - Return BulkDeleteResultDto { DeletedCount, Success, ErrorMessage }

**BulkDeleteResultDto.cs**
```csharp
public record BulkDeleteResultDto(
    int DeletedCount,
    bool Success,
    string? ErrorMessage = null,
    IReadOnlyList<string>? FailedTransactionIds = null
);
```

### Layer 3: Infrastructure — Repository Method

**SupabaseTransactionRepository.DeleteTransactionsByIdsAsync()**
- Input: UserId, IEnumerable<TransactionId> ids
- Query:
  ```sql
  DELETE FROM transactions
  WHERE user_id = @userId
    AND id IN @ids
  RETURNING id
  ```
- Returns: Count of deleted rows
- Error handling: Catch Postgrest exceptions → propagate to handler

### Layer 4: Frontend — UI Components

**Index.cshtml Updates**
1. Add checkbox column to transaction table
   - Each row gets `<input type="checkbox" class="transaction-checkbox" data-transaction-id="@transaction.Id" />`
2. Add table header checkbox for "Select All"
   - Toggles all visible transaction checkboxes
   - Shows X selected message (e.g., "3 selected")
3. Add "Delete Selected" button
   - Disabled by default (enabled via JavaScript when ≥1 checkbox checked)
   - On click: Show confirmation modal

**Index.cshtml.cs Updates**
- Add handler method: `async Task OnPostBulkDeleteAsync()`
  - Extract checked TransactionIds from form submission
  - Dispatch: `await _mediator.Send(new BulkDeleteTransactionsCommand(userId, transactionIds))`
  - Handle response: Show success toast or error message
  - Refresh transaction list

**_BulkDeleteConfirmationModal.cshtml (New)**
- Modal title: "Confirm Delete"
- Modal body: "Are you sure you want to delete {Count} transaction(s)? This action cannot be undone."
- Buttons: Cancel (dismiss modal), Confirm (submit form)
- ESC key support: Close modal without deleting

**JavaScript (wwwroot/js/transactions.js)**
- Toggle "Delete Selected" button enabled/disabled based on checked count
- Sync "Select All" checkbox with individual transaction checkboxes
- Show/hide confirmation modal
- Display success toast on deletion

## Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Cross-tenant data deletion | Handler validates UserId match before deletion (loaded transaction set must equal requested set) |
| Concurrent deletes lose selection | Selection state stored client-side only; server validates each delete independently |
| No rollback if partial delete fails | Delete wrapped in database transaction; Postgrest client handles rollback |
| User confusion on delete | Confirmation dialog shows exact count; success toast confirms completion |
| Performance on large deletes | Limit 1000 transactions max; Postgrest handles IN clause efficiently |
